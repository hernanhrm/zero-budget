# Fix: Sign-in Redirect Loop Issue

## Problem
After signing in, user is redirected to `/` but quickly gets sent back to `/sign-in`. This is a session persistence issue.

## Root Cause
1. **Race condition**: `signIn.email()` uses `callbackURL: window.location.origin` which triggers a redirect, but code also calls `navigate({ to: "/" })` - conflicting redirects
2. **Missing cookie configuration**: Auth client has no explicit cookie settings, causing session cookies to not be properly persisted

## Files to Modify

### 1. `frontend/apps/app/src/features/auth/sign-in/sign-in-form.tsx`
**Change:** Remove manual `navigate()` call - let `callbackURL` handle the redirect

```typescript
// BEFORE (race condition)
const { error } = await authClient.signIn.email({
  email: value.email,
  password: value.password,
  callbackURL: window.location.origin,  // Redirects to /
  rememberMe: true,
})
// ... if no error
navigate({ to: "/" })  // <-- CONFLICTING REDIRECT

// AFTER - Remove navigate call, use callbackURL only
const { error } = await authClient.signIn.email({
  email: value.email,
  password: value.password,
  callbackURL: "/",  // Direct path for redirect
  rememberMe: true,
})
// Remove the navigate() call from success handler
```

### 2. `frontend/apps/app/src/lib/auth-client.ts`
**Add:** Cookie configuration for proper session persistence

```typescript
export const authClient = createAuthClient({
  baseURL: import.meta.env.VITE_IDENTITY_URL ?? "http://localhost:8081",
  cookieName: "better-auth.session",
  cookie: {
    sameSite: "lax",
    path: "/",
    secure: import.meta.env.PROD,
  },
})
```

### 3. `backend/cmd/identity/src/auth.ts` (if needed)
**Check:** Ensure `trustedOrigins` includes the frontend origin and add cookie prefix

```typescript
trustedOrigins: (
  process.env.TRUSTED_ORIGINS ?? "http://localhost:3000"
).split(","),
advanced: {
  cookiePrefix: "zero-budget",
},
```
